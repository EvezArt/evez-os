# Mesh Config Fix - 2026-06-27 17:51 UTC

## Problem
4 of 6 GCP nodes were crash-looping on gateway startup with "agents.defaults: Invalid input"

## Root Cause
`agents.defaults` had an orphan `fallbacks` key at the top level. The schema uses `additionalProperties: false`, so any unrecognized key = validation failure.

The fallbacks were supposed to be INSIDE `model`:
```json
"model": {
  "primary": "vultr/zai-org/GLM-5.1-FP8",
  "fallbacks": ["groq/llama-3.3-70b-versatile", ...]
}
```
But they were also being written as `agents.defaults.fallbacks` (top-level) which is NOT a valid schema key.

## Fix Applied
1. Removed orphan `agents.defaults.fallbacks` from all affected nodes
2. Kept `agents.defaults.model.fallbacks` (inside the model dict - this IS valid per schema)
3. Reset systemd rate-limiters on nodes that had hit restart backoff limits
4. Added same fallback chain to Vultr-KNOT for mesh consistency

## Known Issue
- openclaw-gcp (136.118.144.227): SSH is firewalled (port 22 closed). Gateway on 18789 is healthy. Need to fix SSH access separately.

## Lesson
- `agents.defaults` has `additionalProperties: false` — any field not in the schema = instant validation crash
- Always check schema with `config.schema.lookup` before adding new keys
- `fallbacks` goes INSIDE `model` as `{primary, fallbacks}`, NOT as a sibling key
